# Frontier Design Pillars

These principles define how Frontier should feel to play.

Whenever a new feature is proposed, it should reinforce these pillars rather than work against them.

---

# 1. Discover the World — Don't Purchase Progress

Technology is never bought.

Technology is discovered.

Players advance by interacting with the world.

Observation, experimentation, and survival should naturally lead to new ideas.

The player should feel like they are inventing solutions rather than unlocking them.

Good examples:

* Discovering Primitive Toolmaking after handling stone and wood.
* Learning to farm after observing wild plants.
* Discovering pottery after working with clay.

---

# 2. Time Is the Most Valuable Resource

Everything meaningful consumes time.

Searching.

Traveling.

Crafting.

Building.

Resting.

The player is not choosing what to do.

They are choosing what deserves their time.

Time connects every system in Frontier.

---

# 3. Civilization Learns — Individuals Practice

Knowledge belongs to civilizations.

Skill belongs to survivors.

If one survivor discovers fire, the civilization remembers.

If another survivor joins later, they benefit from civilization discoveries but still develop their own personal skills through practice.

---

# 4. Every Tool Changes the World

A new tool should unlock new possibilities.

The Stone Axe should not simply gather wood faster.

It should allow players to interact with things that were previously impossible.

Every crafted tool should meaningfully expand gameplay.

---

# 5. Systems Create Stories

The game should avoid scripted progression whenever possible.

Interesting stories should emerge from systems interacting.

Examples:

* A storm delays travel.
* Hunger forces an early return.
* A discovery changes future decisions.
* A harsh winter alters resource priorities.

Players should remember experiences rather than quests.

---

# 6. Data Before Code

Content should exist as data whenever practical.

Items.

Recipes.

Locations.

Actions.

Discoveries.

Adding new gameplay should usually mean creating resources rather than modifying engine code.

---

# 7. Simplicity Creates Depth

Individual systems should remain easy to understand.

Depth should come from multiple simple systems interacting.

Avoid solving problems by adding unnecessary complexity.

Prefer combinations of existing systems over entirely new mechanics.

---

# 8. Every Action Has Consequences

Actions should always matter.

Searching advances time.

Crafting consumes resources.

Travel creates opportunity costs.

Every meaningful choice should influence future decisions.

---

# 9. Information Should Be Clear

The player should understand why something happened.

If an action cannot be performed, explain why.

If a discovery occurs, explain what caused it.

Hidden calculations are acceptable.

Hidden rules are not.

---

# 10. Build for Expansion

Every major system should be designed with future growth in mind.

New content should fit naturally into existing systems.

The goal is not to predict every future feature.

The goal is to avoid rebuilding foundations.

---

# The Frontier Test

Before implementing a feature, ask:

* Does this reinforce discovery?
* Does it interact with time?
* Does it create meaningful decisions?
* Does it build upon existing systems?
* Could this be implemented as data instead of code?
* Will this create interesting stories?

If most answers are "yes," the feature probably belongs in Frontier.
